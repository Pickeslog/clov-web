# Clov 프론트엔드 작업 절차

이 문서는 **팀원이 이슈 하나를 맡아 머지까지 가는 전 과정**을 정한다. 순서를 지키면 4명이 동시에 작업해도 충돌하지 않는다.

## 1. 시작 전에 읽는다

1. **이슈 본문 전체** — 특히 **"범위 밖 (하지 말 것)"** 항목. 여기 적힌 파일은 다른 사람이 같은 시간에 만지고 있다.
2. `AGENTS.md` — 이 저장소의 규칙
3. `docs/API-CONTRACT.md` — API 계약. **화면이 쓰는 응답 필드는 반드시 여기서 확인**한다. 추측하지 않는다.
4. 화면 작업이면 `../web-design-repository/screen-spec-source/` 의 해당 화면 명세

이슈에 안 적힌 걸 "겸사겸사" 고치지 않는다. 별개 이슈로 남긴다.

## 2. 브랜치

```
git checkout main
git pull origin main
git checkout -b feat/<이슈번호>-<작업이름>
```

- `main`에서 직접 작업하지 않는다
- **`git pull`을 건너뛰지 않는다.** 다른 사람 PR이 먼저 머지됐으면 그 위에서 시작해야 나중에 충돌이 안 난다
- 브랜치 이름: `feat/` (기능·개선) · `fix/` (버그) · `chore/` (설정·문서)

## 3. 코드 규칙

### 반드시 지킬 것

| 규칙 | 이유 |
|---|---|
| API 호출은 `src/api/` 모듈을 통해서만 | 컴포넌트에서 `fetch`/`axios` 직접 호출 금지 |
| 응답은 `data`가 바로 온다 | `src/api/client.js`가 `{success, data}` 봉투를 이미 벗긴다. `res.data.data` 같은 코드 금지 |
| 목록은 `data?.items ?? []` | 목록 응답은 `{ items: [...] }` 봉투다 |
| ID는 **문자열** | 서버가 문자열로 준다. `===` 비교 시 숫자와 섞지 말 것 |
| 로딩 판정은 **`isPending`** | react-query v5에서 `isLoading`은 재시도 대기 중 `false`라 `data`가 없는데 렌더돼 크래시난다. **실제로 터졌던 버그다** |
| 시각은 `src/lib/datetime.js` 사용 | 백엔드는 오프셋 없는 UTC(`2026-07-24T00:13:04`)를 준다. `Z`를 붙여 파싱해야 한국 시각으로 맞다. 직접 `new Date(값)` 하지 말 것 |
| 날짜 파싱 함수는 **값이 없어도 안 죽게** | 타임스탬프 하나 때문에 화면 전체가 죽은 사고가 있었다 |

### 하지 말 것

- **방장·관리자·대표자 같은 권한 개념 도입** — 우정공간의 모든 멤버는 동등하다. 서비스의 핵심 원칙이라 예외 없다
- **계약(`API-CONTRACT.md`)을 임의로 바꾸기** — 백엔드 응답이 이슈와 다르면 먼저 이슈에 댓글로 물어본다
- **이슈 "범위 밖" 파일 수정** — 리팩터링 중이라도
- 남이 만든 파일을 "정리하는 김에" 손대기

### 리팩터링 이슈일 때

**합격 기준은 "화면이 전과 똑같은 것"이다.** 코드를 한 곳으로 모으는 작업이지 동작을 바꾸는 게 아니다. 화면 문구·레이아웃·계산 결과가 달라지면 잘못된 것이다.

## 4. 끝내기 전 검증

```
npm --prefix clov-web run lint
npm --prefix clov-web run build
```

둘 다 통과해야 한다. 그다음 **실제로 화면을 열어본다**:

1. 개발 서버(5173)에서 내가 고친 화면에 들어간다
2. 이슈에 적힌 확인 항목을 눌러본다
3. 브라우저 콘솔(F12)에 **빨간 에러가 없는지** 확인한다

빌드가 통과해도 실행하면 터지는 버그가 있다. 반드시 눈으로 본다.

## 5. PR

- PR 본문 **첫 줄에 `Closes #<이슈번호>`** — 안 쓰면 이슈가 자동으로 안 닫혀서 수동으로 닫아야 한다
- 제목은 `feat: 무엇을 했는지 (#이슈번호)` 형식
- 본문에는 **무엇을 · 왜 · 어떻게 검증했는지**를 쓴다. "수정함" 한 줄은 리뷰가 불가능하다
- **CI가 초록불이 될 때까지 기다린 뒤** 리뷰를 요청한다
- 머지는 리더가 한다

## 6. 막혔을 때

혼자 30분 이상 헤매지 않는다. **이슈에 댓글로** 상황을 적는다.

- 무엇을 하려 했는지
- 무슨 일이 일어났는지 (에러 메시지 그대로 복사)
- 뭘 시도해봤는지
